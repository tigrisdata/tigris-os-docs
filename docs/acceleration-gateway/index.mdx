# TAG — Tigris Acceleration Gateway

TAG is a high-performance S3-compatible caching proxy for
[Tigris object storage](https://www.tigrisdata.com). It sits between your
application and Tigris, transparently caching objects on local NVMe storage to
deliver sub-millisecond read latency while preserving full S3 API compatibility.

<div className="mermaid-frame">
```mermaid
flowchart LR

    classDef tag fill:#4a148c,stroke:#ce93d8,color:#fff

    classDef upstream fill:#006064,stroke:#80deea,color:#fff

    A[Your App]:::app -->|GET /bucket/key| T[TAG]:::tag
    T --> V[Validate SigV4 locally]:::tag
    V --> C[(RocksDB cache on NVMe)]:::cache
    C -->|HIT| R[Return response<br/>X-Cache: HIT]:::tag
    C -->|MISS| U[Tigris Object Storage]:::upstream
    U -->|Stream response| T
    T -->|Cache + return<br/>X-Cache: MISS| A

````
</div>

TAG requires no application code changes: point your S3 client at TAG instead of Tigris, and reads are automatically accelerated. TAG is distributed as pre-built binaries and Docker images for Tigris customers.

## Why TAG

Object storage is durable and scalable, but every read crosses the network. For latency-sensitive workloads (ML model serving, real-time analytics, and high-throughput data pipelines), that round-trip adds up fast.

TAG eliminates it for repeated reads by maintaining a local RocksDB-backed cache that serves objects in microseconds instead of milliseconds. TAG is not a generic HTTP cache. It understands S3 natively: it coalesces concurrent requests for the same object into a single upstream fetch, serves range requests from cached full objects, revalidates stale entries with conditional requests, and invalidates cache entries when objects are written or deleted.

> **Bottom line:** One endpoint change. Zero code changes. Sub-millisecond reads.

## Performance

Benchmarked on AWS (c6in.16xlarge client, i3en.24xlarge server with NVMe storage):

| Object Size | Ops/sec | Bandwidth | TTFB p50 |
|---|---:|---:|---|
| 1 KiB | 75,717 | 73.9 MiB/s | < 1 ms |
| 100 KiB | 33,350 | 3.2 GiB/s | < 1 ms |
| 1 MiB | 10,955 | 10.7 GiB/s | Single-digit ms |
| 4 MiB | 2,775 | 10.8 GiB/s | Single-digit ms |

TAG saturates a 100 Gbps NIC at 85+ Gbps for 1 MiB+ objects, with CPU utilization at only about 12%, leaving significant headroom for additional workloads.

:::note

These numbers reflect cache-warm steady state. First requests for uncached objects incur a round-trip to Tigris.

For full methodology, environment specs, and complete `warp` and `go-ycsb` results, see [TAG Benchmarks](https://github.com/tigrisdata/tag-deploy/blob/main/docs/benchmarks.md).

:::

## Key Features

### S3-Compatible API

TAG implements the S3 REST API for commonly used Tigris operations. Clients interact with TAG using standard AWS SDKs, the AWS CLI, or any S3-compatible library. The only change is the endpoint URL.

:::warning[Path-Style Addressing Only]

TAG supports path-style addressing only (`http://host:port/bucket/key`). Virtual-hosted style is not supported. Configure your S3 client accordingly.

:::

### Transparent Proxy Mode

In default mode, TAG preserves the client's original AWS SigV4 Authorization header and forwards requests to Tigris as-is. TAG adds cryptographically signed proxy headers so Tigris can validate both the client's identity and TAG's identity in a single round-trip.

:::info[Security]

TAG never sees or stores client secret keys. Clients authenticate directly with Tigris using their own credentials.

:::

### Embedded Cache with RocksDB

Each TAG node runs an embedded cache backed by RocksDB, optimized for NVMe storage. Objects are stored using a two-key pattern:

| Key | Stored data |
|---|---|
| `meta|my-bucket|photo.jpg` | `Content-Type`, `ETag`, `Content-Length`, `Last-Modified` |
| `body|my-bucket|photo.jpg` | Raw object bytes |

Separating metadata from body enables efficient HEAD requests and conditional operations without reading the full object.

### Request Coalescing

When multiple clients request the same uncached object simultaneously, TAG coalesces them into a single upstream fetch:

### Range Request Optimization

TAG serves byte-range requests from cached full objects when available. On a
range cache miss, TAG does two things in parallel: serves the requested range
directly from Tigris, and triggers a background fetch of the full object. Future
range requests for any byte range are then served entirely from cache.

:::tip

This is especially useful for ML workloads that access model weights with random-access patterns. The first range request warms the full object into cache.

:::

### Write-Through Invalidation

PutObject, DeleteObject, DeleteObjects (bulk), and CopyObject operations
automatically invalidate corresponding cache entries before forwarding to
Tigris. A tombstone mechanism prevents in-flight background cache writes from
resurrecting deleted objects.

:::caution

Objects modified directly on Tigris (bypassing TAG) remain cached until the TTL expires (default 60 minutes) or a client forces revalidation with `Cache-Control: no-cache`. Route writes through TAG for immediate consistency.

:::

For deployments requiring more cache capacity than a single node provides, TAG
nodes can form a distributed cache cluster:

### Request Flows

**Cache hit:** Client -> TAG validates SigV4 locally -> cache lookup -> returns
object with `X-Cache: HIT`. Tigris is not contacted.

**Cache miss:** Client -> TAG forwards to Tigris -> streams response to client
while writing to cache -> `X-Cache: MISS`. Next request is a hit.

**Coalesced request:** Multiple clients GET the same uncached object -> first
becomes the fetcher -> others attach as broadcast listeners -> all receive data
simultaneously -> one upstream request.

**Range request:** Client sends `Range: bytes=0-1023` -> if cached, serve from
cache -> if not, serve range from Tigris plus background fetch of full object ->
future ranges served from cache.

### Ports

| Port | Protocol   | Purpose                                |
| ---- | ---------- | -------------------------------------- |
| 8080 | HTTP/HTTPS | S3 API (client-facing)                 |
| 7000 | TCP        | Memberlist gossip (cluster discovery)  |
| 9000 | TCP        | gRPC (cache key routing between nodes) |

## S3 API Coverage

TAG supports commonly used S3 operations, validated against 214 tests from the
`ceph/s3-tests` compatibility suite.

| Category          | Operations                                                                                                               |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------ |
| Objects           | Get, Head, Put, Delete, Copy, Tagging, ACL                                                                               |
| Buckets           | List, Create, Delete, Head, ListV1/V2, Location, Versioning, ACL, Policy, CORS, Tagging, Lifecycle, DeleteObjects (bulk) |
| Multipart Uploads | Initiate, UploadPart, UploadPartCopy, Complete, Abort, ListParts, ListMultipartUploads                                   |
| Auth              | SigV4 headers, presigned URLs, chunked encoding                                                                          |

:::warning[Not Supported]

Object versioning, server-side encryption (SSE-C/S3/KMS), Object Lock, POST object (browser uploads), multi-range requests, and virtual-hosted style addressing.

:::

## Cache Behavior

TAG adds an `X-Cache` response header to every response:

| Value         | Meaning                                                          |
| ------------- | ---------------------------------------------------------------- |
| `HIT`         | Served from cache (includes successful revalidations)            |
| `MISS`        | Fetched from Tigris, now cached                                  |
| `REVALIDATED` | Revalidated with Tigris; object changed and new content returned |
| `BYPASS`      | Cache skipped (client sent `Cache-Control: no-store`)            |
| `DISABLED`    | Caching turned off server-side                                   |

**What gets cached:** HTTP 200 responses within the size threshold (default 1
GiB) that do not include `Cache-Control: no-store` or `private`.

**What does not:** Non-200 responses, objects above the size threshold, and
objects with cache-prevention headers.

**Client cache control:**

| Header                     | Behavior                               |
| -------------------------- | -------------------------------------- |
| `Cache-Control: no-cache`  | Revalidate with Tigris before serving  |
| `Cache-Control: max-age=0` | Same as `no-cache`                     |
| `Cache-Control: no-store`  | Bypass cache entirely and do not store |

## Security Model

### Transparent Proxy Authentication

<div className="mermaid-frame">
```mermaid
sequenceDiagram
    participant C as Client
    participant T as TAG
    participant S as Tigris
    C->>T: Signed request (SigV4)
    T->>S: Forward original auth + proxy headers
    S->>S: Validate client signature
    S->>S: Validate TAG proxy signature
    S-->>T: Response + derived signing keys
    T-->>C: Response (internal headers stripped)
```
</div>

TAG never sees client secret keys. Proxy headers are signed with HMAC-SHA256 so
Tigris can verify TAG's identity alongside the client's.

### Local Authentication for Cache Hits

After a client's first request, TAG learns the client's derived SigV4 signing
keys from Tigris (encrypted with AES-256-GCM, decrypted locally). Subsequent
requests are validated without contacting Tigris.

| Component       | Purpose                   | Capacity     | TTL                   |
| --------------- | ------------------------- | ------------ | --------------------- |
| DerivedKeyStore | SigV4 signing keys        | 100K entries | 48 hours              |
| AuthzCache      | Per-bucket auth decisions | 1M entries   | 10 min (configurable) |

:::warning[Important]

Authorization is per-bucket. Access to one bucket does not grant access to others. Entries are revoked immediately on a Tigris 403 response.

:::

### Credential Requirements

TAG requires its own Tigris credentials (`AWS_ACCESS_KEY_ID` /
`AWS_SECRET_ACCESS_KEY`) with read-only access to all buckets in the same
organization as clients it serves. These are used for proxy header signing and
background cache operations.

### Endpoint Validation

TAG validates the upstream endpoint at startup and allows connections only to:

| Pattern         | Example                          |
| --------------- | -------------------------------- |
| `localhost`     | `http://localhost:8080`          |
| `*.tigris.dev`  | `https://fly.storage.tigris.dev` |
| `*.storage.dev` | `https://t3.storage.dev`         |

Any other endpoint is a fatal startup error. This prevents TAG from being
pointed at arbitrary S3-compatible services.
